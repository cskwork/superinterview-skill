System Design Interview / Model Answer

대규모 음식 주문·배달 플랫폼 설계

읽기 비중이 높은 탐색 경로는 CDN/캐시/검색 색인으로 최적화하고, 주문·결제는 강정합성과 멱등성으로 보호한다. Kafka가 서비스를 분리하고 트래픽 스파이크를 흡수한다.

strong vs eventual idempotency geospatial dispatch graceful degradation ~45 min

01요구사항 정의

설계 전에 범위를 좁힌다. Coupang Eats / Uber Eats급 고트래픽 음식 주문·배달 플랫폼. 핵심 흐름: 고객이 식당을 탐색 → 주문 → 결제 → 식당 수락 → 라이더 배정 → 실시간 배달 추적.

기능적 요구사항

  • 고객: 식당 탐색, 메뉴 조회, 주문, 결제, 주문 상태 추적
  • 식당: 메뉴 관리, 주문 수락/거절, 조리 상태 갱신
  • 라이더: 배달 요청 수신, 배정 수락, 픽업, 배달 진행 갱신
  • 알림, 프로모션, 환불, CS 이벤트 지원

비기능적 요구사항

  • 고가용성, 확장성, 스파이크 내성 (점심/저녁/공휴일/할인)
  • 주문·결제 = 강정합성; 식당 탐색·배달 추적 = 결과적 정합성 허용
  • 탐색/추적은 저지연; 주문·결제는 초저지연보다 신뢰성 우선
초기 스케일 가정
  • DAU ~1M
  • 피크 식당 탐색 읽기: 10k-50k QPS
  • 피크 주문 제출: 1k-5k 건/분
  • 낮은 수치를 주더라도 10배 성장을 전제로 설계

03전체 아키텍처

시스템을 여러 서비스로 분리한다: API 게이트웨이, 사용자, 식당, 메뉴, 검색, 주문, 결제, 배차/라이더 매칭, 배달 추적, 알림, 프로모션/쿠폰, 이벤트 스트리밍, 데이터 분석·모니터링.

  • 트랜잭션 서비스(주문/결제): ACID가 필요하므로 MySQL/PostgreSQL. 고스케일에선 user_id/region/order_id로 샤딩
  • 비동기 워크플로: Kafka로 이벤트 분리 → 결합도 감소, 스파이크 회복력 향상

04핵심 주문 플로우

가장 중요한 흐름은 주문 배치다.

  1. 고객이 식당·메뉴 탐색 - 읽기 비중. Redis 캐시/검색 색인에서 우선 서빙, 미스 시 DB → 캐시 재채움.
  2. 주문 제출. Order Service가 검증: 사용자, 식당 상태, 메뉴 가용성, 가격, 쿠폰, 주소, 배달권역. 주문 생성, 상태 PENDING_PAYMENT.
  3. Payment Service 결제 처리. 멱등해야 함 - 클라이언트/서버 생성 idempotency key로 중복 결제 방지.
  4. 결제 성공 → PaymentSucceeded 이벤트 Kafka 발행 → Order Service가 소비해 상태 PAID로.
  5. 식당 앱/서비스가 주문 수신. 수락 시 RestaurantAccepted 이벤트 발행.
  6. Dispatch Service가 위치, 라이더 가용성, 거리, 조리 시간, 배달 SLA 기반 라이더 매칭. 배정 시 RIDER_ASSIGNED.
  7. Delivery Tracking이 라이더 앱의 GPS 갱신 수신. 고객에게 WebSocket/SSE/푸시로 준실시간 갱신.

06캐싱 전략

  • 식당/메뉴 데이터: Redis 캐시. TTL 1-5분 (결제 데이터만큼 엄격하지 않음).
  • 사용자 세션/토큰: 빠른 접근을 위해 Redis.
  • 레이트리밋: 사용자/IP/기기별 요청 수 Redis 추적.
  • 핫 식당/프로모션: 이벤트 시작 전 캐시 사전 워밍(pre-warm).
주의

Redis를 주문/결제의 진실 공급원(source of truth)으로는 쓰지 않는다. 주문·결제는 내구성 있는 관계형 DB에 저장한다.

07DB & 정합성 전략

도메인마다 다른 정합성 모델을 적용한다.

  • 강정합성: 주문 생성, 결제, 취소, 환불 - 돈과 사용자 신뢰에 직결.
  • 결과적 정합성: 식당 검색, 메뉴 표시, 배달 추적, 추천 랭킹 - 약간의 stale은 감내.

트랜잭션 DB는 읽기 확장/페일오버를 위해 복제 사용(primary=쓰기, replica=읽기). 트래픽 증가 시 region/order_id로 샤딩 (예: 서울/부산 분할, 또는 order_id에 region+timestamp 인코딩).

핵심 인덱스

orders(user_id, created_at) · orders(restaurant_id, created_at) · orders(status, updated_at) · delivery_tasks(rider_id, task_status) · payments(order_id) · payments(idempotency_key)

08딥다이브: 배차 & 라이더 매칭

배차는 배달 시스템에서 가장 중요한 부분 중 하나다. 라이더 위치, 식당/고객 위치, 라이더 가용성, 현재 작업량, 예상 조리 시간, 약속 배달 SLA를 모두 고려해 매칭한다.

  • 지역 질의: Redis GEO, PostGIS, 또는 H3/geohash 위치 인덱싱.
  • 라이더는 연속적으로 GPS를 보내지만, 모든 갱신을 DB에 쓰면 쓰기 부하 폭발 → 최근 위치는 Redis/고처리량 위치 저장소, 중요 마일스톤만 DB 영속화.
  • 단순 알고리즘: 반경 내 가용 라이더 탐색 → 예상 픽업 시간 순 정렬 → 최적 라이더 배정 → 오퍼 전송. 타임아웃 내 미수락 시 다음 라이더로 재시도.
  • 대규모: 배칭, 예상 조리 시간 예측, 라이더 공정성, 배달 비용, 지역별 수요-공급 불균형까지 고려.

09트래픽 스파이크 처리

점심/저녁/공휴일/할인 캠페인에 스파이크가 예상된다. 여러 계층으로 흡수한다:

  1. CDN + Redis로 백엔드 읽기 압력 감소.
  2. API 게이트웨이 레이트리밋/스로틀.
  3. Kafka가 비동기 처리를 위해 버스트 흡수 (알림, 분석, 식당 이벤트 로그, 일부 배차 재시도).
  4. 서비스 수평 오토스케일 (CPU, 메모리, 지연, 큐 lag, 에러율 기반).
  5. 비핵심 기능 강등: 부하 시 개인화 추천 비활성, 추적 갱신 빈도 감소, 식당 랭킹 단순화.
핵심 원칙

핵심 주문·결제는 안정적으로 유지하고, 비핵심 기능은 강등 가능해야 한다.

10장애 처리

주요 장애 시나리오
  • 결제 성공 but 주문 상태 갱신 실패 → Kafka 재시도 + 정합 작업이 최종 수정. 결제 provider 기록과 내부 기록을 주기적 비교.
  • 식당 무응답 → 타임아웃 후 자동 취소, 고객 환불.
  • 라이더 배정 실패 → Dispatch가 더 넓은 반경으로 재시도 또는 수동 운영으로 에스컬레이션.
  • Kafka 지연 → 중간 상태를 사용자에게 명확히 노출 ("결제 완료, 식당 확인 대기 중").
  • Redis 다운 → DB 풀백, 단 레이트리밋으로 DB 보호.
  • DB primary 장애 → 복제본 + 자동 페일오버. 다중 AZ 배포 필수.

11트레이드오프 분석

  • 정합성 vs 가용성: 결제/주문상태/취소/환불 = 강정합성. 검색/랭킹/메뉴 캐시/배달 추적 = 가용성·성능을 위해 결과적 정합성.
  • 지연 vs 신뢰성: 탐색·검색은 캐시·색인으로 저지연. 주문 배치는 정확성·멱등성·신뢰성을 위해 약간의 추가 지연 감내.
  • 비용 vs 성능: 핫/빈번 데이터만 캐시. 읽기 비중과 쓰기 비중 서비스를 분리해 독립 스케일.
  • 모놀리스 vs 마이크로서비스: 작으면 너무 많은 서비스로 시작하지 않는다. 단 Coupang 규모면 Order/Payment/Dispatch/Restaurant/Search/Notification 분리가 합리 (서로 다른 스케일링 패턴·장애 경계).

12병목 & 관측가능성

주요 병목

  • 주문/결제 DB → 인덱싱, 읽기 복제본, 샤딩, 짧은 트랜잭션.
  • 피크 시 식당/메뉴 탐색 → CDN, Redis 캐시, 사전 워밍, 검색 색인 최적화.
  • 배차 매칭 → 지역 인덱싱, 지역 분할, 비동기 매칭 워크플로.
  • 알림 fan-out → Kafka + 워커 풀로 비동기.
  • 외부 결제 provider 지연/장애 → 타임아웃, 백오프 재시도, 멱등 키, 정합 작업.

관측

  • 지표: API p50/p95/p99, 서비스별 에러율, 주문 성공률, 결제 성공/실패율, Kafka consumer lag, 배차 배정 지연, 배달 SLA 위반, DB CPU/슬로우쿼리/락 대기, Redis 히트율·메모리, 외부 결제 지연.
  • 분산 트레이싱: 한 주문이 게이트웨이 → Order → Payment → Kafka → Dispatch → Notification을 가로지르는 추적.
  • 인시던트 대응: 대시보드, 알림, 런북, 기능 플래그 (장애 중 비핵심 기능 빠른 비활성).

13요약

읽기 비중 서비스와 트랜잭션 중심 서비스를 분리한다. 식당 탐색·메뉴는 CDN/Redis/검색 색인으로 최적화. 주문·결제는 강정합성·멱등성을 갖춘 관계형 DB. Kafka가 서비스 결합을 풀고 스파이크를 흡수. 배차는 지역 인덱싱과 비동기 매칭. 부하 하에서 우아하게 강등하면서 가장 중요한 흐름(성공적 주문·배달)을 보호한다.

핵심 트레이드오프

주문/결제 워크플로의 정확성·신뢰성을 최우선하고, 검색/추적/추천/알림은 결과적 정합성을 감내한다. 이로써 플랫폼이 스케일하면서도 사용자 신뢰를 유지한다.
